01 —

Redis as a Data Structure Server

Redis is widely used as a cache, but the design intent was broader: a data structure server. Caching became the dominant use case because Redis's core design properties — in-memory storage, fast lookups, rich types — fit caching workflows perfectly.

Redis vs Memcached

PropertyMemcachedRedis
Value typeOpaque blobTyped structure (string, hash, list, set, sorted set…)
Server awarenessDumb store — no ops on valueServer understands and mutates types directly
Round-tripsFetch → deserialize → modify → serialize → storeSingle atomic server-side command (LPUSH, ZADD, HINCRBY)
Use case fitSimple key/blob cacheCache, leaderboards, queues, pub/sub, rate limiting…
Key insight: Because Redis understands the value type, operations happen on the server atomically. You don't round-trip data to the client just to modify it.
02 —

Single-Threaded Execution Model

Redis command execution is single-threaded. This is a deliberate design choice, not a limitation.

Why single-threaded works

For in-memory operations, the bottleneck is never CPU execution — a hash lookup takes nanoseconds. The real bottleneck is the network round-trip (microseconds). Since CPU isn't the constraint, adding threads for execution would only introduce lock contention without improving throughput.

Latency hierarchy
RAM access ~100ns  →  Command execution (hash lookup etc.) ~few hundred ns  →  Network round-trip ~100µs–1ms

Network dominates. Single-threaded execution is free atomicity at nearly zero cost.

What single-threading gives you

Every command is atomic by construction. INCR, LPUSH, multi-step Lua scripts — no locks needed. The moment you parallelize execution, every compound operation requires synchronization. Redis avoids this entirely.

Event loop — not one thread per connection

Redis uses epoll/kqueue to multiplex thousands of client connections onto a single event loop. It is not one OS thread per client. This is why Redis sustains 100k+ ops/sec with a single thread.

03 —

Network I/O Threading (Redis 6+)

At high connection counts, reading bytes off sockets and writing responses became the actual bottleneck — not command execution. Redis 6 added I/O threads to parallelize exactly this, while keeping the execution model intact.

Request flow

Client Request
     │
     ▼
I/O Threads        ← read bytes off socket, parse RESP protocol
     │
     ▼
Main Thread        ← execute command against in-memory data (sequential)
     │
     ▼
I/O Threads        ← serialize response, write bytes back to socket
     │
     ▼
Client Response
Clean separation: I/O threads own the wire. Main thread owns the data. They never share data structures — no locks anywhere in the hot path.

Coordination mechanism

This is not a typical thread pool with a shared concurrent queue. The main thread explicitly assigns clients to each I/O thread's dedicated list, then busy-polls an atomic counter until all I/O threads finish. It spins rather than blocking on a condition variable — burning a few CPU cycles to avoid kernel context-switch overhead. A deliberate tradeoff at microsecond latency targets.

Main Thread
├── assigns client → I/O Thread 1's list
├── assigns client → I/O Thread 2's list
└── assigns client → I/O Thread 3's list
        │
        ▼  (I/O threads work in parallel)
        │
Main Thread busy-polls counter → 0
        │
        ▼
Main Thread executes all queued commands sequentially
        │
        ▼
Main Thread assigns responses back to I/O threads

RESP parsing happens in I/O threads

I/O threads don't just move raw bytes — they parse the Redis Serialization Protocol (RESP). The main thread receives clean, structured command objects. It never touches wire bytes.

Why not fully multi-thread execution?

Because execution being single-threaded is what preserves atomicity guarantees. Parallelizing execution would require locks on every shared data structure, trading simplicity and correctness for marginal CPU gains on a workload that was never CPU-bound.

Supported types

TypeTypical use caseExample command
StringCounters, cached valuesSET / GET / INCR
HashObject fields (user profile)HSET / HGET / HINCRBY

Dual encoding — the hidden optimization

For small collections, Redis does not use the "real" data structure internally. It uses a compact flat encoding called listpack (formerly ziplist) — a contiguous byte array with no pointers. This is cache-line friendly and has dramatically lower per-element memory overhead.

Encoding promotion — one-way
Compact encoding (listpack) → used when collection is small (e.g. hash < 128 entries, entries < 64 bytes)
Full data structure (hash table, skiplist) → promoted to when threshold is crossed

Promotion is automatic and transparent. It is one-way — a promoted structure never reverts. This is unlike a dynamic array like List<T> which grows in place; Redis switches the underlying representation entirely. 
Small hash (≤128 keys)       Large hash (>128 keys)
┌─────────────────────┐      ┌──────────────────────┐
│ listpack             │  →   │ hash table            │
│ [k1|v1|k2|v2|k3|v3] │      │ bucket[] → entry →   │
│ contiguous memory    │      │ next (chained)        │
└─────────────────────┘      └──────────────────────┘
  cache-friendly, zero         pointer overhead,
  pointer overhead              O(1) avg lookup
Why this matters: In production, a huge fraction of Redis keys are small. Listpack encoding means Redis memory efficiency is significantly better than naive implementations — without any changes to the API.

Persistence — RDB & AOF

Redis is an in-memory store. If the server process dies, all data in RAM is lost — unless persistence is configured. Redis offers two complementary mechanisms.

RDB — Point-in-time Snapshots

Periodically dumps the entire dataset to a binary file (dump.rdb). Configurable trigger — e.g. "save if 1000+ keys changed in last 300s". On restart, Redis loads the RDB file and reconstructs the full in-memory state.

t=0          t=300s       t=600s       t=750s (crash)
 │             │             │               │
 ▼             ▼             ▼               ✗
[snapshot]  [snapshot]  [snapshot]     ← 150s of writes lost
How RDB snapshots work without blocking
Redis calls fork(). The child process writes the snapshot to disk; the parent keeps serving requests. The OS uses copy-on-write — parent and child share memory pages until the parent modifies a page, at which point only that page is copied. Consistent snapshot, zero blocking on the hot path.

AOF — Append Only File

Every write command is appended to a log as it executes. On recovery, Redis replays the entire log from top to bottom to reconstruct state. This is a WAL (Write-Ahead Log) pattern.

SET foo bar
INCR counter
LPUSH mylist a
HSET user:1 name "Alice"
...

Three fsync policies control the durability/performance tradeoff:

PolicyBehaviorMax data loss
alwaysfsync on every writeZero (theoretically)
everysecfsync every second (default)~1 second
noOS decides when to flushOS buffer worth

AOF Rewrite — log compaction

AOF grows unboundedly. INCR counter called a million times is a million log entries, but final state is one key. Redis compacts the log periodically via AOF rewriting — it rewrites the log to represent only current state, discarding redundant history.

Production setup — both together

SetupRecovery speedData loss risk
RDB onlyFast (binary load)Up to snapshot interval
AOF onlySlow on large datasets (replays all commands)~1 second (everysec)
RDB + AOFFast (load RDB, replay AOF delta only)~1 second
Honest caveat: Even with both enabled, Redis gives best-effort durability — not guaranteed durability. The design philosophy accepts this in exchange for performance. Redis is not a source of truth; pair it with a durable store (Postgres, Cosmos DB) for data you cannot afford to lose.
06 —

Replication — Full & Partial Sync

Redis replication is primary → replica, asynchronous by default. The replica is a read-only copy. Writes always go to primary; replicas serve reads.

Async tradeoff: Primary does not wait for replicas to acknowledge before confirming a write to the client. Primary can acknowledge a write, then crash before replication completes. That write is lost.

Primary availability during sync

The primary remains fully available for reads and writes during a full sync. This is the same fork() trick used by RDB snapshots — the child process streams the snapshot to the replica while the parent continues serving requests normally. New writes during the sync window are buffered in the replication backlog and streamed to the replica once it finishes loading the RDB.

Primary
   │
   ├── fork() ──► Child streams RDB snapshot to replica
   │                   (seconds for large datasets)
   │
   └── Parent keeps serving reads + writes
              ↓
         buffers new writes in replication backlog

Full sync — PSYNC

Triggered on first connect, or when the replica has fallen too far behind. Replica sends PSYNC ? -1. Primary forks, generates an RDB snapshot, streams it. After the replica loads it, the primary replays the backlog delta to bring the replica up to present state.

Partial sync — the ring buffer

Every primary maintains two tracking values and a buffer:

ConceptWhat it is
Replication IDA unique identity string for this primary's dataset lineage — like a UUID for the command history
Replication offsetA monotonically increasing byte counter — increments with every byte streamed to replicas. Acts as a cursor into the command stream
Replication backlogA fixed-size circular ring buffer of recent write commands. Default 1MB. New writes overwrite oldest entries once full
 
Replication backlog (ring buffer, e.g. 1MB)

oldest surviving         current primary
     offset                   offset
        │                        │
        ▼                        ▼
[...commands........commands........commands]
        ↑
  gets overwritten as new writes arrive

On reconnect after a brief disconnect, the replica says: "My replication ID is X, and I was at offset N. Give me everything after N." The primary checks whether offset N is still inside the ring buffer.

Replica last offset: 3000
Backlog window:      5000 → 9000 (current)

3000 < 5000 → offset overwritten → full sync required

─────────────────────────────────────────────

Replica last offset: 6500
Backlog window:      5000 → 9000 (current)

6500 is within window → stream delta 6500→9000 only → partial sync ✓
Key condition: If the replica's last offset falls before the oldest entry still in the ring buffer, the data it needs is gone — overwritten by newer writes. Full sync is the only option. If the offset is within the backlog window, only the delta is streamed.

Backlog size — an operational concern

If your primary is write-heavy and a replica disconnects even briefly, the backlog can fill and overwrite old offsets — forcing an expensive full sync. Tuning repl-backlog-size to cover your expected disconnection window × write throughput is a real production consideration. Getting this wrong turns routine network blips into repeated full syncs.

Replication ID change on failover

When a replica is promoted to primary (via Sentinel failover), it gets a new replication ID. Replicas following the old primary now have a mismatched ID and cannot partial sync against the new primary — they must full sync. Redis 4+ addresses this with a secondary replication ID: the new primary remembers its old ID for a window, allowing already-caught-up replicas to partial sync even after a failover.

07 —

Sentinel — Automated Failover

Replication alone gives you a hot standby but not automatic failover. If the primary dies, someone has to manually promote a replica. Sentinel solves this — it is a separate process that monitors your Redis topology and acts on failures.

Sentinel 1 ─┐
Sentinel 2 ─┼──► monitors ──► Primary + Replicas
Sentinel 3 ─┘

Three jobs

JobWhat it does
MonitoringPings primary and replicas periodically to detect failures
NotificationAlerts operators or other systems when something looks wrong
Automatic failoverPromotes the most caught-up replica if primary is confirmed down

Quorum — why odd numbers

A single sentinel declaring the primary down is not enough. A configurable quorum of sentinels must agree before failover triggers — this prevents false positives from a single sentinel losing network visibility.

Always run 3 or 5 sentinels, never 2. With 2, one failing leaves you deadlocked with no majority. With 3, one can die and you still have quorum.

3 sentinels, quorum = 2

Sentinel 1: primary is down ─┐
Sentinel 2: primary is down ─┴── quorum reached → start failover
Sentinel 3: unreachable         (not needed)

Failover sequence

StepWhat happens
1. SDOWNA sentinel stops getting responses — marks primary subjectively down
2. ODOWNSentinels gossip — quorum agrees → marked objectively down
3. Leader electionSentinels elect a leader among themselves (Raft-like vote)
4. Replica selectionLeader picks the most caught-up replica by replication offset
5. PromotionSends REPLICAOF NO ONE to promote it to primary
6. ReconfigurationOther replicas are told to follow the new primary
7. Client updateSentinel updates service discovery so clients find the new primary

Client pattern — don't hardcode the primary address

During failover, there is a window where the topology is in flux. Clients that cached the old primary address will fail. The correct pattern is to use a Sentinel-aware client that queries Sentinel for the current primary address on each new connection — never hardcoding the primary IP directly.

Sentinel is not a proxy. It does not sit in the data path. It only manages topology and tells clients where the primary is. The client still connects directly to Redis.
08 —

Redis Cluster & Hash Slots

Single-node Redis (even with Sentinel) is bounded by one machine's RAM and write throughput. Cluster breaks the dataset across multiple primary nodes — horizontal sharding with built-in replication and failover.

Hash slot partitioning

Redis Cluster uses 16384 hash slots. Every key maps to exactly one slot via CRC16(key) % 16384. Slots are distributed across primary nodes.

Node A  →  slots 0     – 5460
Node B  →  slots 5461  – 10922
Node C  →  slots 10923 – 16383

Every node knows the full slot map. If a client hits the wrong node, it gets a MOVED redirect pointing to the correct node. Smart cluster clients cache the slot map locally and route correctly on the first attempt.

Client → SET user:42 "alice" → Node A
Node A computes slot 7423 → owned by Node B
Node A replies: MOVED 7423 node-b:6379
Client retries directly against Node B

Why 16384 slots specifically

Cluster nodes gossip constantly — heartbeat messages include the sender's slot map as a bitmap so every node stays aware of the full topology.

16384 slots → 16384 bits → 2048 bytes → 2KB per heartbeat
65536 slots → 65536 bits → 8192 bytes → 8KB per heartbeat
16384 was the sweet spot — enough slots for granular distribution across thousands of nodes, small enough to gossip cheaply. With even 1000 nodes that's ~16 slots per node, plenty of resharding granularity.

Built-in replication — no Sentinel needed

Each primary shard has one or more replicas. Sentinel is not used — the cluster nodes monitor each other directly via peer-to-peer gossip on the cluster bus (port +10000). When enough nodes agree a primary is unreachable, the cluster promotes its replica automatically. Same quorum principle, different actors.

Primary A  ←→  Replica A
Primary B  ←→  Replica B       ← minimum production setup: 3+3 nodes
Primary C  ←→  Replica C
SetupWho votesMinimum nodes
SentinelExternal Sentinel processes3 Sentinels + primary + replicas
ClusterPrimary nodes themselves3 primaries + 3 replicas

Multi-key operations — what breaks

Any operation touching multiple keys only works if all keys map to the same slot — same node. Redis cannot coordinate a single atomic operation across multiple nodes.

MSET user:1 "alice"  user:2 "bob"

CRC16("user:1") % 16384 → slot 5474 → Node B
CRC16("user:3") % 16384 → slot 1625 → Node A  ← different node

→ ERROR: CROSSSLOT keys not in same slot
CaseResult
All keys → same node✓ Works, fully atomic
Keys → different nodes✗ CROSSSLOT error
Keys → mid-migration slot⚠ ASK redirect (temporary)

Hash tags — forcing co-location

Wrap a substring in {} — Redis only hashes the part inside the braces, forcing related keys onto the same slot.

user:{42}:profile   →  CRC16("42") % 16384  ─┐
user:{42}:settings  →  CRC16("42") % 16384  ─┼── same slot, same node ✓
user:{42}:sessions  →  CRC16("42") % 16384  ─┘
Hot slot risk: Co-locating keys intentionally means one entity's data all lands on one node. If that entity dominates traffic, one node bears disproportionate load — undermining the distribution benefit. Design hash tags carefully.

Resharding — MOVED vs ASK

Adding or removing nodes requires slots to migrate. Redis does this live without downtime. During migration a slot is in transition — clients get two different redirects with distinct semantics.

MOVED  →  slot permanently lives here now, update your slot map cache
ASK    →  slot is mid-migration, try this node just for this one request

Key design shift

Single-node Redis never required thinking about where a key lives. Cluster makes locality a first-class design concern — which keys need to be co-located for multi-key operations (use hash tags), which are genuinely independent (let them distribute naturally). Design keys around access patterns, not just naming conventions.

Eviction Policies

Redis is bounded by the machine's RAM. When usage crosses maxmemory, every new write triggers an eviction attempt before the write is accepted. If Redis can't free memory, it rejects writes with an OOM error while still serving reads.

The eight policies

ScopePolicyWhat it evicts
No evictionnoevictionRejects writes. Never evicts.
TTL keys onlyvolatile-lruLRU among keys with a TTL set
volatile-lfuLFU among keys with a TTL set
volatile-ttlSoonest-to-expire among TTL keys
volatile-randomRandom among keys with a TTL set
All keysallkeys-lruLRU across entire keyspace
allkeys-lfuLFU across entire keyspace
allkeys-randomRandom across entire keyspace

LRU vs LFU — the key distinction

LRU (Least Recently Used) tracks last_access_time per key. Evicts the key with the oldest timestamp. No concept of how many times it was accessed — only when it was last touched.

LFU (Least Frequently Used) tracks an access counter per key with time decay applied. The counter decays if the key isn't accessed — so old frequency doesn't count forever. Makes frequency reflect recent relevance, not lifetime total.

Key A   accessed 1000x last month, 0x this week
Key B   accessed 50x this week

LRU → evicts A (oldest last_access_time)
LFU → evicts A (counter decayed, low recent frequency)
LRU asks: when did you last show up?
LFU asks: how often have you been showing up lately?

LRU struggles with scan patterns — a one-time bulk read of cold keys updates their timestamps, potentially evicting genuinely hot keys. LFU is immune because a single access barely moves the frequency counter.

Approximate LRU — not exact

True LRU requires a global ordered structure tracking access time for every key — expensive at Redis's scale. Redis uses approximate LRU instead: samples a random subset of keys (default 5, tunable via maxmemory-samples), evicts the least recently used among that sample.

Full keyspace: millions of keys
                    │
                    ▼
Sample 5 random keys → evict the LRU among those 5

Not exact — the globally least-recently-used key may not be in the sample. Increasing maxmemory-samples improves accuracy at the cost of CPU per eviction.

Policy selection guide

Use caseRecommended policy
Pure cache, all keys disposableallkeys-lru or allkeys-lfu
Mix of persistent + cached keysvolatile-lru (only evict TTL keys)
Keys have meaningful TTLsvolatile-ttl
Never lose data silentlynoeviction (handle OOM in app)
Hot keys accessed repeatedly long-termallkeys-lfu

TTL expiry — distinct from eviction

Eviction is memory pressure driven. TTL expiry is time driven — Redis proactively removes keys whose time has passed via two complementary mechanisms running in parallel.

MechanismHow it worksTradeoff
Lazy expiryKey checked on access — deleted if expiredZero background cost, but expired keys linger until touched
Active expiryBackground job periodically samples TTL keys, deletes expired onesReclaims memory even for abandoned keys never accessed again

Quorum & Distributed Systems

Quorum is not Redis-specific — it is a fundamental principle in any distributed system that needs to make a collective decision: leader election, failover, write acknowledgement. Once the mental model clicks, it surfaces everywhere.

The universal rule

Total nodes      = N
Quorum           = N/2 + 1  (strict majority)
Fault tolerance  = N/2      (nodes that can fail while system stays alive)
NodesQuorum neededCan tolerateNote
220 failuresUseless for HA
321 failureMinimum viable
431 failureSame as 3, one node wasted
532 failuresRecommended for critical systems
Always deploy odd numbers. Even numbers give you no extra fault tolerance but cost more. 4 nodes tolerates the same 1 failure as 3 nodes.

Odd nodes solve split brain

Split brain is what happens when a network partition causes two halves of a cluster to each think they are the primary and independently accept writes. Data diverges with no way to reconcile.

Quorum prevents this — but only odd node counts guarantee it cleanly.

5 nodes, network partition

Side A: 3 nodes → strict majority of 5 → elects leader ✓
Side B: 2 nodes → minority → refuses to elect, goes silent ✓

One leader. Split brain impossible.
4 nodes, network partition

Side A: 2 nodes → 2/4, not majority → can't elect
Side B: 2 nodes → 2/4, not majority → can't elect

Both sides halt. No leader. Cluster unavailable.
Odd nodes guarantee that any partition always produces one side with strict majority — that side safely elects, the other stays silent.

Even nodes can produce a symmetric 50/50 partition with no majority anywhere — leading to full halt or split brain depending on the implementation.

Halting is safer than diverging

A cluster refusing to elect a leader means downtime — recoverable. Two leaders accepting conflicting writes means data corruption — often unrecoverable. Quorum enforces the safer failure mode.

Where quorum appears across systems

SystemQuorum used for
Redis SentinelSentinel processes voting to declare primary ODOWN
Redis ClusterPrimary nodes voting to declare a shard primary FAIL
etcd / ZookeeperDistributed coordination — Kubernetes control plane uses etcd with min 3 nodes
Kafka ISRWrite committed only when enough in-sync replicas confirm it
Cosmos DBWrite acknowledged after quorum of regional replicas confirm — consistency levels tune how strict this is
SQL Server Always OnAutomatic failover requires quorum of availability group members + witness node
Raft consensusAlgorithm underpinning etcd, CockroachDB, TiDB — majority quorum for leader election and log commits

CAP theorem connection

During a network partition, a system must choose between two guarantees — it cannot have both simultaneously.

ChoiceBehaviourExamples
CP — Consistent + Partition tolerantRefuses writes until quorum is reachable. Downtime over divergence.Redis Cluster, etcd, Cosmos strong consistency
AP — Available + Partition tolerantAccepts writes on both sides, reconciles later. Divergence over downtime.Cosmos eventual consistency, DynamoDB
Neither is wrong
CP vs AP is a business decision — can your system tolerate data divergence, or downtime? Financial systems typically choose CP. High-availability read-heavy systems often choose AP. The right answer depends on what failure mode hurts your users less.
