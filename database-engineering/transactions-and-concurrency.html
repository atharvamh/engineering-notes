ACID, Transactions and Concurrency

Module I: Transactions & ACID

s1. What is a Database Transaction?

A transaction is a collection of one or more database operations (SQL queries like SELECT, INSERT, UPDATE, DELETE) executed as a single, logical unit of work.

A transaction has two possible outcomes:

  • Commit: All changes are permanently saved to the database.
  • Rollback (Abort): If any operation fails, all changes made during the transaction are undone, leaving the database exactly as it was before the transaction started.

📌 Transaction Demarcation

Demarcation Modes

Database transactions are initiated and managed in two primary modes:

  • Explicit Transactions: Boundaries are manually declared by the engineer using explicit state commands (e.g., BEGIN TX, COMMIT, ROLLBACK).
  • Implicit Transactions (Autocommit): The database engine automatically packages standalone queries into individual single-statement transactions execution loops.

s2. ACID Guarantees Deep Dive

To guarantee data integrity, database management systems (DBMS) must enforce the ACID properties. Let’s break down exactly what each letter means and how the database engine implements it.

🛑 A - Atomicity ("All or Nothing")

Atomicity guarantees that a transaction is treated as a single, indivisible unit. Either the entire transaction succeeds, or the entire transaction fails. There is no such thing as a "half-committed" transaction.

The Problem

Imagine transferring $100 from Account A to Account B. The DB updates Account A (subtracts $100), but the server crashes before it updates Account B. If the database leaves it like this, $100 vanishes into thin air.

The Database Solution

Write-Ahead Logging (WAL) / Undo Logs: Before the database modifies any actual data on disk, it writes the intended change to an append-only log file (the WAL). If a transaction fails mid-way, the engine reads the Undo Log to reverse the partial changes and restore the old values.

🔄 C - Consistency ("Follow the Rules")

Consistency ensures that a transaction can only bring the database from one valid state to another, maintaining all predefined schema rules, constraints, and invariants.

The Problem

A database has a rule (constraint) that an account balance cannot drop below $0. A transaction tries to withdraw $200 from an account that only has $100.

The Database Solution

The database explicitly validates schema constraints (e.g., NOT NULL, UNIQUE, FOREIGN KEY, CHECK constraints) during execution. If any constraint is violated, the database aborts the transaction.

Note: Consistency also relies heavily on Atomicity and Isolation to maintain business-level correctness.

🔒 I - Isolation ("Don't Step on My Toes")

In a production database, thousands of transactions run concurrently. Isolation ensures that the concurrent execution of transactions leaves the database in the same state as if they were executed sequentially (one after the other). It prevents transactions from seeing each other's intermediate, uncommitted data.

The Problem

Transaction 1 updates a row but hasn't committed yet. Transaction 2 reads that row. Transaction 1 then encounters an error and rolls back. Transaction 2 has now read "dirty data" that technically never existed.

The Database Solution
  • Locking Mechanisms: The database places locks on data rows or tables (Shared locks for reading, Exclusive locks for writing) to control access.
  • Multi-Version Concurrency Control (MVCC): Modern databases (like PostgreSQL and MySQL InnoDB) create "snapshots" of the data. When you read, you read a historical version of the data from when your transaction started, ensuring writers don't block readers, and readers don't block writers.

⚠️ Concurrency Read Phenomena

When isolation controls relax to increase performance throughput, the database can experience three primary read anomalies:

  • Dirty Reads: A transaction reads raw uncommitted data written by another in-flight transaction that might eventually roll back.
  • Non-Repeatable Reads: A transaction reads the same row twice within its execution lifespan but finds mutated data because another transaction committed an update in between.
  • Phantom Reads: A transaction executes a range query (e.g., finding accounts with balances > $50) twice but finds new "phantom" rows injected by another transaction that committed an insert in between.

💾 D - Durability ("Survive the Apocalypse")

Durability guarantees that once a transaction has been committed, its changes are permanently recorded in non-volatile storage (like an SSD or HDD). Even if the entire operating system crashes or the server loses power a millisecond after the commit, the data will not be lost.

The Problem

Writing data directly to the database data files on disk is slow. To speed things up, databases temporarily hold changes in volatile RAM (the OS page cache). If power cuts out, anything in RAM vanishes.

The Database Solution

fsync() system call: When a transaction commits, the database forces the OS to flush the transaction's log records from RAM to physical, non-volatile disk using an OS command called fsync. The transaction is not considered "committed" until the disk controller confirms the WAL bytes are safely written to physical media.

s3. Durability & Recovery Techniques

A database's survival under unexpected power grid failure, operating system crash, or physical server reboot depends entirely on its Durability implementation. Writing data directly to complex, indexed database pages on disk for every single write is highly inefficient because it requires expensive random I/O. Modern engines bypass this with smart logging and serialization strategies.

💾 Three Core Durability Patterns

1. Write-Ahead Logging (WAL)

Before any in-memory modification is made to an index or data page (Buffer Pool), the changes are appended sequentially to a transaction log on disk (the WAL). Sequential disk writes are exponentially faster than random page updates on disk, allowing databases to optimize write latency.

The Recovery Routine: Redo and Undo

When a database crashes and reboots, it runs a two-step recovery protocol:

  • Redo Phase (Roll Forward): The engine plays the WAL forward to write all changes from committed transactions that had not yet been flushed from volatile RAM buffer pool to the actual tablespace disk pages.
  • Undo Phase (Roll Back): The engine reads Undo logs or WAL transaction boundaries to identify transactions that were active (in-flight) at the moment of the crash but never committed, completely reversing their partial writes to maintain strict Atomicity.
2. Asynchronous Snapshots

Instead of continuous modification logging, the database takes a full, point-in-time binary dump of the entire database state and saves it to non-volatile storage asynchronously at structured intervals (e.g., every 5 minutes).

The Durability Gap & Copy-on-Write

Taking a snapshot of a huge database is resource-heavy. On Linux, engines like Redis use the fork() system call to spin off an isolated child process that writes the snapshot dump while utilizing Copy-on-Write (CoW). This lets the parent process continue serving traffic concurrently without blocking reads/writes.

The Risk: If the database crashes 4 minutes after the last snapshot, all updates written during those 4 minutes are completely lost. This is known as the Snapshot Durability Gap.

3. Append-Only File (AOF)

Maintains a continuous ledger of every single write command received. Unlike binary WAL files (which log physical page changes), AOF is a logical, command-based log (e.g., logging raw Redis commands like SET user:1 alice).

Fsync Tuning Trade-offs

How durable is an AOF? It depends on the flushing frequency configuration:

  • fsync everysec (Default): Appends command to memory, flushes to disk once per second. High throughput; potential to lose at most 1 second of data on crash.
  • fsync always: Forces an fsync() on the disk controller for every write query. Maximum durability, but drops performance to the speed of magnetic/flash write cycles.
  • fsync no: The database appends to the OS buffer cache and lets the OS default kernel scheduler (usually every 30 seconds) handle disk flushing. Highest performance, lowest durability.

⚡ The Hybrid Durability Solution (Snapshots + AOF)

While Asynchronous Snapshots and Append-Only Files (AOF) each have severe architectural drawbacks when run independently, their combination represents a masterclass in modern database durability. Redis implements this as its primary production-ready persistence strategy.

How the Hybrid Strategy Works

By pairing a static, point-in-time snapshot with a lightweight transactional command ledger, the database successfully mitigates both of their worst limitations:

  • Bridging the Snapshot Durability Gap: Instead of leaving a 5-minute data-loss gap between snapshots, the database constantly appends ongoing mutations to the AOF. On crash, you lose practically nothing (or at most 1 second of data if using fsync everysec).
  • Solving the Slow AOF Recovery: Under a pure AOF system, recovering a year-old database requires running millions of sequential commands from scratch, taking hours. In the hybrid layout, the AOF is systematically truncated/compacted to only contain the incremental command mutations written since the last snapshot.
The Under-The-Hood Architecture (RDB-AOF Hybrid Preamble)

Modern engines (such as Redis 4.0+) merge these files directly during a background rewrite cycle:

  1. The engine spawns a background child process via fork() to rewrite the append-only file.
  2. The child process writes the current database state directly to the beginning of the file as a compact, binary RDB snapshot (the Preamble).
  3. Any new write operations that occur concurrent to the write are appended to the end of the same file as logical, human-readable AOF command logs.

On Recovery: The database engine opens the file, loads the binary RDB preamble instantly to reconstruct 99.9% of the database state in milliseconds, and then plays back the tiny remaining sliver of logical AOF logs to bring the system up-to-date. Durability is maximized, and Recovery Time Objective (RTO) drops to sub-seconds.

📝 Physical Logging vs. Logical Logging

A major design choice in database logging is whether to log physical or logical modifications. This choice dictates the recovery speed and log size:

Logging Type What is Recorded Pros Cons Examples
Physical Logging (WAL) Specific byte-level adjustments within physical page blocks (e.g., "Set byte 42 to 0x05 on page 4"). Fast, deterministic recovery. Safe against partial page writes. Generates massive log sizes. Hard to replicate across database versions. PostgreSQL WAL, MySQL InnoDB Redo Log
Logical Logging (AOF) The exact operations/commands executed (e.g., "Insert Alice into Users" or "SET user:1 alice"). Log sizes are significantly smaller. s4. Lost Updates (Write-Write Anomalies)

While Read Phenomena deal with what transactions are allowed to see, the Lost Update is a devastating write-write anomaly. It occurs when two transactions execute a Read-Modify-Write sequence concurrently on the exact same row.

The Problem: Step-by-Step Collision

Imagine Account A has a starting balance of $100.

  • T1: Reads balance ($100). Wants to deposit $50.
  • T2: Reads balance ($100). Wants to deposit $20.
  • T1: Local calculation is $150. Writes balance = 150 to database and commits.
  • T2: Local calculation is $120. Writes balance = 120 to database and commits.

Result: The balance is written as $120. T1's deposit of $50 is completely wiped out (lost).

The Engineering Solutions

Engineers solve the Lost Update using three primary patterns:

  • 1. Pessimistic Locking (Exclusive Row Locks)

    Explicitly lock the row during the read stage so no other transaction can access it until you are finished.

    SELECT balance FROM accounts WHERE id = 1 FOR UPDATE;

    Effect: T2 is blocked from reading the row until T1 commits or aborts.

  • 2. Optimistic Concurrency Control (OCC) / CAS

    Maintain a version or updated_at column on the row. Verify the version hasn't changed at write time.

    UPDATE accounts SET balance = 120, version = 2 WHERE id = 1 AND version = 1;

    Effect: If T1 updated the version to 2 first, T2's update statement will target 0 rows, prompting T2 to abort and retry.

  • 3. MVCC Snapshot Isolation (First-Committer-Wins)

    In advanced isolation levels (such as Repeatable Read in PostgreSQL), the engine actively detects concurrent updates to the same row. If a collision is found, the database instantly aborts the second writer with a serialization error:

    ERROR: could not serialize access due to concurrent update

s5. Transaction Isolation Levels

The ANSI SQL-92 standard defines four classical transaction isolation levels based on which read anomalies they prevent. However, modern database engines implement these levels using vastly different internal architectures.

📶 The Four Classic Levels + Snapshot Isolation

1. Read Uncommitted (Dirty Reads Allowed)

The absolute lowest tier of isolation. A transaction can instantly view raw, uncommitted writes from other in-flight transactions.

Engine Support Reality

Most modern relational databases (such as PostgreSQL) do not actually support Read Uncommitted. If you request it, PostgreSQL silently and automatically upgrades your connection's isolation level to Read Committed.

2. Read Committed (Strictly Committed Reads)

Transactions can only read data that has been permanently committed. This successfully prevents Dirty Reads.

The Implementation Detail

Each SQL query inside the transaction receives a new read-snapshot (or locks are acquired and released statement-by-statement). Because snapshots are generated per statement, a transaction running multiple queries can see data change between queries (allowing Non-Repeatable Reads).

3. Repeatable Read (No Value Mutation Mid-Transaction)

Guarantees that any row read by a transaction will return the exact same values for the entire duration of that transaction.

Is this a locking mechanism?

Historically: Yes. In traditional databases, this was achieved via Two-Phase Locking (2PL). The database placed a Shared Lock on every read row and held it until the transaction committed. This completely blocked concurrent writers from modifying those rows, severely damaging write throughput.

Modern Database Reality: No! Modern systems (like PostgreSQL and MySQL InnoDB) avoid locking readers entirely. Instead, they implement Repeatable Read using MVCC snapshots. When a reader reads, it queries historical tuple versions, allowing writers to continue writing concurrent updates without any blocking.

4. Snapshot Isolation (Point-in-Time Database Snapshots)

An MVCC-specific isolation pattern. When your transaction starts, the database freezes a logical "snapshot in time" representing the exact state of the entire database. Your transaction executes its queries against this local, static view of the world.

Engine Implementation Detail

PostgreSQL uses Snapshot Isolation to implement its SQL-standard Repeatable Read isolation level. Under Postgres, this level is so powerful that it actually prevents Phantom Reads as well!

5. Serializable (Sequential-Equivalent Execution)

The highest isolation standard. Transactions behave as if they were executed in a strict, single-threaded queue (sequentially).

Implementation Strategies: Literal sequential execution would destroy hardware parallelism. Relational engines optimize this using:

  • MySQL (InnoDB): Uses Next-Key Locking (pessimistic range locks on index blocks) to stop concurrent transactions from inserting rows into query ranges.
  • PostgreSQL: Uses Serializable Snapshot Isolation (SSI). It allows transactions to run concurrently without locks but builds a dependency graph of writes and reads in memory. If it detects a potential cycle that would break serial order, it aborts the offending transaction with a serialization failure.

⚙️ Why Database Implementations Vary

The standard SQL-92 definitions were written when Two-Phase Locking (2PL) was the only popular database design pattern. Because of this, the standard defined isolation levels purely by which anomalies were blocked. When MVCC engines emerged, they bypassed those lock boundaries entirely, resulting in major platform variations:

Database System Read Uncommitted Handling Repeatable Read Behavior Serializable Strategy
PostgreSQL Silently upgraded to Read Committed. Uses Snapshot Isolation. Blocks Dirty, Non-Repeatable, AND Phantom Reads! SSI (Serializable Snapshot Isolation) — optimistic, lock-free anomaly detection.
MySQL (InnoDB) Supported. Allows true dirty reads. Uses Snapshot Isolation (Read View). Allows Phantom Reads unless locked explicitly with FOR UPDATE. Next-Key Locking — pessimistic index range locking.
Oracle Not supported at all. Not supported directly. Users are directed to use Serializable (which is actually Snapshot Isolation). Snapshot Isolation (first-committer-wins write checks).

s6. MVCC Under the Hood (The Update Lifecycle)

Modern relational engines (like PostgreSQL and MySQL InnoDB) avoid reader-writer blocking entirely using Multi-Version Concurrency Control (MVCC). Under MVCC, updating a record is physically implemented as an INSERT of a new record version coupled with a logical deletion of the old record version. The database maintains multiple physical copies of the same row simultaneously to serve active, long-running queries.

The 5 Readers vs. 1 Writer Concurrency Scenario

Suppose Row A initially holds an account balance of $100. Let's trace how 5 active reading transactions interact with 1 writing transaction modifying Row A concurrently:

  • The Readers (Tx 1–5): These 5 transactions are active and executing analytical scans. Because they started before Tx 6 commits, their transactional snapshots freeze Row A's state at $100.
  • The Writer (Tx 6): This active transaction executes: UPDATE accounts SET balance = 150 WHERE id = A; and commits.
  • The Concurrency Separation: Rather than physically overwriting the bytes of Row A, the database engine keeps both versions on disk. Tx 1–5 read the consistent old_a ($100) copy, while Tx 6 and any new transactions read the committed new_a ($150) version. **No transaction blocks, and lock latency is zero.**

The MVCC Storage Clutter Penalty (Dead Tuples)

When running transactional updates under MVCC, old row versions are not deleted in-place immediately. This introduces major performance side-effects:

  • Table Bloat: Hundreds of thousands of dead tuples sit in the physical pages, increasing file size and slowing down sequential page scans.
  • Write Amplification: Retaining old snapshots forces indexes to store redundant mappings, consuming storage write overhead.
The Remediation: Postgres Vacuum & InnoDB Purging

Database engines continuously clean historical records using isolated background processes:

  • PostgreSQL VACUUM: Spawns background worker processes that scan tables, identify rows where xmax is older than the oldest active transaction, mark their slot pointers as reusable, and release the block space back to the table heap.
  • MySQL InnoDB Purge Thread: Periodically discards old undo log records and clears off-page overflow pointers, keeping indexes tightly compacted.
Engine Comparisons: PG Heap vs. InnoDB Undo Logs

Relational storage architectures utilize vastly different approaches to maintaining multi-version isolation:

  • PostgreSQL (In-Heap Multi-Versioning): Writes new row copies directly into the main table heap pages. Both active and inactive row data sit together. Excellent simplicity; heavy vacuuming overhead.
  • MySQL InnoDB (Out-of-Heap Undo Logging): Overwrites the main page row in-place, but writes the old record bytes to a dedicated side-log space called the Undo Tablespace. If a reader needs an old version, it traverses the rollback pointers and reconstructs the historical row in RAM on-the-fly. Main table files stay incredibly clean; requires CPU cycles for old row reconstructions.

s7. Non-Repeatable vs. Phantom Reads

A deep point of engineering detail is the dividing line between Non-Repeatable Reads and Phantom Reads. Under the traditional SQL-92 standard, these anomalies occur at different isolation thresholds due to the physics of standard database locks.

The Fundamental Difference: Row vs. Range

The distinction boils down to whether the concurrent transaction mutates existing rows or introduces completely new matching rows:

  • Non-Repeatable Reads (Existing Row Mutations): Transaction A reads a row. Transaction B concurrently updates or deletes that exact same physical row and commits. When Transaction A reads it again, the values have changed.
  • Phantom Reads (Range Alterations): Transaction A executes a range query (e.g., WHERE salary > 5000) and finds 3 matching rows. Transaction B concurrently inserts a brand new row with a salary of 6000 and commits. When Transaction A runs the exact same query again, a 4th "phantom" row suddenly appears.

🔒 Why Repeatable Read Fails at Phantoms (Two-Phase Locking)

Under a traditional Two-Phase Locking (2PL) database engine, isolation is maintained using physical row locks. This architecture exposes a major structural flaw when dealing with inserts:

The Lock Blind Spot

Suppose Transaction A runs a range query: SELECT * FROM users WHERE age > 30;.

  • The database engine successfully places a Shared Lock (S) on every single user row currently returned where age is greater than 30.
  • This lock prevents concurrent transactions from running UPDATE or DELETE queries on those locked rows. This completely prevents Non-Repeatable Reads.
  • The Catch: Transaction B can still run INSERT INTO users (name, age) VALUES ('Eve', 35); because there was no physical user record for 'Eve' for Transaction A to lock! The row is newly injected, bypasses the locks, and appears as a phantom.
The Solution: Next-Key Locking vs. MVCC Snapshots

Database engines solve this blind spot in two distinct ways depending on their architecture:

  • 1. Next-Key Locking (Pessimistic - MySQL InnoDB):

    Instead of locking individual rows, the database index engine locks the "gaps" (empty range intervals) between index keys. If Transaction A scans the range for age > 30, the engine blocks other transactions from inserting any new values into those index slots, stopping phantoms in their tracks.

  • 2. MVCC Snapshot Isolation (Optimistic/Lock-free - PostgreSQL):

    Under MVCC, Transaction A is running against a static, frozen snapshot of the database state. Since the snapshot does not change when Transaction B commits a new insert, the inserted rows are naturally filtered out of Transaction A's range results. Under PostgreSQL, setting your isolation level to Repeatable Read naturally blocks Phantom Reads without using any row locks.

s9. Consistency: Data (ACID) vs. Reads (CAP)

A classic point of confusion in database engineering is the word "Consistency". The industry overloads this term, using it to describe two fundamentally distinct concepts depending on the architectural scope: Consistency in Data (Single-Node ACID) and Consistency in Reads (Distributed Systems CAP).

The Core Confusion

Many engineers assume that the "C" in ACID and the "C" in CAP mean the exact same thing. In reality, they address totally different scopes of system stability:

  • ACID "C" (Consistency in Data): Deals with application/schema rules on a single local instance.
  • CAP "C" (Consistency in Reads): Deals with the speed and uniformity of synchronizing data states across global geographical regions.

📊 Structural Differences

Metric Consistency in Data (ACID) Consistency in Reads (CAP)
Primary Scope Single Database Instance (standalone database logic). Distributed Clusters (multiple database nodes across the globe).
Core Objective Protect data integrity rules and schema constraints. Ensure all users see the exact same data value at the exact same instant.
Enforcing Agent The DBMS Schema Engine (Validators, constraints, triggers) & Application Business Logic. Replication/Consensus Protocols (Raft, Paxos, Quorums, Sync/Async replication).
Typical Threat Erroneous applications writing invalid values, breaking invariants. Network partitions, speed-of-light propagation latency (replication lag).
Guarantees NOT NULL, UNIQUE, Foreign Keys, CHECK(balance >= 0) constraints. Linearizability (Strong Consistency), Read-Your-Own-Writes, Eventual Consistency.

🔒 1. Consistency in Data (ACID "C") — Local Constraints

This is a data-validity property. It means transitioning a database instance from one valid, constraint-compliant state to another. If a transaction attempts to write data that breaks your constraints, the DBMS physically rejects the transaction and rolls it back, keeping the data clean.

Single Instance Example

Suppose you declare a table constraint: CHECK (active_users <= maximum_capacity).

If a single-node PostgreSQL instance runs a transaction that tries to increment the user count past maximum capacity, the database engine aborts the query, throws a constraint exception, and forces a rollback. The data remains consistent because the rules of the database schema were never allowed to break.

🌐 2. Consistency in Reads (CAP "C") — Global States

This is a distributed-state uniformity property (often called Linearizability). In a modern application, a database is rarely deployed as a single standalone server. Rather, it is deployed as primary and replica nodes spanning multiple geographical regions to minimize latency and provide failover protection.

When you write to a primary node in the US-East region, that write must travel across physical cables to update a replica in EU-Central. Since the speed of light limits how fast data can travel, a client reading from EU-Central a fraction of a millisecond later might read old, stale data. This is Replication Lag.

Distributed Read Collision Scenario

Imagine a distributed system with a Primary Node in New York and a Read Replica in London using asynchronous replication:

  • Step 1: User in New York changes their username to "Alice". The write is accepted by the Primary Node.
  • Step 2: Before New York can replicate the change to London, a User in London queries the London Replica for the same username.
  • Step 3: The London Replica returns the old name: "Bob".

Result: We have a Read Inconsistency. The system has violated CAP "Consistency" because different clients see different values at the exact same point in time.

Architectural Configurations for Read Consistency

To mitigate this, database architects tune replication protocols based on business needs:

  • Synchronous Replication: The Primary waits for all replica instances to acknowledge the write before returning a success message to the client. This guarantees strong read consistency but dramatically spikes write latency.
  • Quorum Writes ($W + R > N$): In NoSQL systems like Cassandra, you configure the cluster so that the number of nodes written to ($W$) plus the nodes read from ($R$) is greater than the total replica nodes ($N$). This mathematically guarantees that a read will always overlap with and retrieve the latest write.
  • Eventual Consistency: Accept that read consistency will be broken temporarily. The system guarantees that replicas will "eventually" catch up and become consistent after replication lag drops to zero.
