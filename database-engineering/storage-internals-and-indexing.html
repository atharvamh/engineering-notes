s9. Table & Heap Internals (Pages & Row IDs)

A relational table is a logical abstraction. Under the hood, the DBMS represents tables as collections of physical files stored in persistent storage (disk). To read, write, or index this data efficiently, databases segment these files into fixed-size processing blocks called Pages and categorize rows inside structures called Heaps.

📦 The Core Elements of Storage Architecture

1. The Page / Block (The I/O Atom)

A Page (or Block) is the absolute atomic unit of database disk I/O. Databases do not read or write individual rows from a drive; instead, they retrieve and commit entire Pages containing those rows (typically 8 KB in PostgreSQL and 16 KB in MySQL InnoDB). Even when executing a simple index query for a single row, the engine pulls the entire physical Page into its volatile memory buffer pool first.

2. Heap Storage vs. Index-Organized Tables

A Heap is an unordered collection of physical pages where data rows (tuples) are written wherever there happens to be free space, completely ignoring physical sort orders. PostgreSQL tables are strict heaps. Conversely, InnoDB tables are Index-Organized (Clustered Indexes), meaning the table's records are physically ordered and stored right inside the primary key's leaf nodes.

3. Physical Row IDs (ctid, TID, DB_ROW_ID)

Every row in a database has an internal, physical address representing its location on disk. This is the Row ID (often called TID or ctid). It is represented as a physical coordinate tuple: (Page Number, Tuple Slot Offset). For instance, a Postgres ctid of (4, 12) means: "Go to Page 4, and read the 12th row pointer." Indexes store these physical Row IDs as leaf values to locate raw table values in a Heap.

The Row Migration Problem (Write Amplification)

When you update a variable-length text field in a row (e.g., from 'Bob' to a long string), the record expands. If the page is full, the modified row can no longer fit in its original slot.

The Risk: If the database physically migrates the row to a new page, it risks breaking every existing index entry on other tables pointing to that row's old physical coordinate, forcing a massive, expensive update of all indexes.

The Indirection Layer: Slotted Page Architecture

Modern relational engines protect against this index fragmentation using a Slotted Page Layout inside every page block:

  • Page Header: Metadata detailing page offsets.
  • Line Pointer Array (Slot Array): Grows forward (left-to-right). Each item is a light index pointing to the actual row offset.
  • Tuple Payloads: Row byte arrays written from the end of the page backward (right-to-left).

The Remedy: If a row moves inside a page, the engine only updates its local Line Pointer. Since the row's physical coordinates index remains unchanged, external index pointers pointing to the Row ID are never broken.

s10. B-Tree & B+Tree Indexing Internals

When a database queries a table without an index, it must execute a full table scan, pulling every single disk page into memory—resulting in severe disk I/O bottlenecks. Indexes solve this by building structured routing tables. In transactional systems, the standard index implementation is the B+Tree.

Why Databases Avoid Binary Search Trees (BST)

Why don't database engines use simple Binary Search Trees (BST) or Red-Black Trees? The reasons are purely physical:

  • Tree Depth & Disk Spanning: A BST has a branching factor of at most $2$. To index $1,000,000$ rows, a BST requires a depth of $\log_2(1,000,000) \approx 20$ levels. Since each parent-child node connection in memory could point to a completely different page on disk, finding a single key would require $20$ random disk reads.
  • Lack of Page Alignment: BST nodes are too small. They store only one key and two child pointers, failing to utilize the $8\text{ KB}$ physical page capacity, which leads to massive memory fragmentation.
The Solution: The B+Tree (Shallow Depth & High Fan-out)

A B+Tree is a self-balancing search tree designed to handle large page-aligned file blocks. It optimizes performance using two distinct node types:

  • Internal Nodes (Routing Guides): These contain only search keys and physical child pointers. They contain no actual data or row pointer payloads. This maximizes the branching factor (fan-out). If a routing page holds 500 keys, its branching factor is $B = 500$.
  • Leaf Nodes (Data Anchors): These store the actual index key values along with their corresponding physical Heap Row IDs (or the actual table rows in a clustered InnoDB table).

The Advantage: By storing all record payloads in the leaf nodes, internal nodes stay lightweight, enabling a massive fan-out. A B+Tree with a branching factor of $B = 500$ can index millions of keys in just $3$ levels—reducing physical disk page lookups to a maximum of $3$ hops!

Key Architectural Variations: B-Tree vs. B+Tree

Most relational engines deploy B+Trees instead of standard B-Trees. Here is why:

  • Standard B-Trees: Store physical records inside both internal and leaf nodes. This drastically decreases the space available for routing keys in internal nodes, increasing the tree depth and requiring more disk reads. Range queries are also slow because the engine must traverse up and down intermediate nodes.
  • Modern B+Trees: Keep data exclusively in the leaf nodes and link all leaf nodes together sequentially via a doubly-linked list. When executing a range scan (e.g., WHERE balance > 100), the engine traverses down to find the starting node, then scans horizontally through the linked leaves, completely bypassing tree traversal overhead.

s11. Clustered vs. Heap Tables (The UUID Write-Amplification Trap)

A critical architectural fork in physical storage engine design is the choice between Heap-Based Storage (used natively by PostgreSQL) and Clustered Indexes / Index-Organized Tables (used natively by MySQL InnoDB). The choice of your table's Primary Key data structure (e.g., auto-incrementing integers vs. random UUIDs) has completely opposite physical performance implications depending on this underlying model.

📊 Structural Architectural Comparison

Storage Property Heap-Based Tables (PostgreSQL Model) Clustered-Index Tables (MySQL InnoDB Model)
Table Physical Ordering Unordered. Data is written to pages sequentially based solely on where free physical space resides. Strictly Sorted. Row payload blocks are organized physically on disk in sorted order of the Primary Key.
Primary Key Leaf Value Contains the Row ID (ctid / TID) — a physical address coordinate pointing to the heap file. Contains the actual complete row data (all table columns are embedded inside the tree).
Secondary Index Leaf Value Contains the physical Row ID (ctid / TID). Directly reads target table page on lookup. Contains the Primary Key value (not a physical disk offset). Forces a double-lookup to fetch row data.
Row Address Stability Volatile. If a row moves inside a page or migrates, its physical Row ID coordinate could theoretically shift. Volatile. Rows physically move across database page blocks during index splits.

The Problem: The Random UUID Clustered-Index Trap

When you insert records with completely random primary keys (such as a UUIDv4) into an InnoDB table, you trigger catastrophic physical write problems:

  • Continuous Page Splits: Because UUIDv4 is completely unsorted, a new row must be written physically in the middle of a random table page on disk. If that targeted 16 KB page is full, the engine must split it, moving 50% of the rows (along with all column payloads) to a new page to clear space.
  • Heavy Random Disk Writes: Instead of writing sequentially to the physical end of the file, the engine must read random, uncached table pages into RAM, write the record, and flush the pages back to disk, causing disk-head thrashing.
  • Massive Secondary Indexes: Because InnoDB secondary indexes store the Primary Key value at their leaf level, if your PK is a wide, 36-character string UUID, your secondary indexes will explode in size, consuming precious RAM.
The Solution: Sequential Keys & Postgres Appends

To mitigate write bottlenecks, relational systems use two structural approaches:

  • 1. Use Sequential Keys for Clustered Tables:

    For engines like InnoDB, use auto-incrementing integers or sorted UUID formats like UUIDv7 (which embeds a millisecond-precision timestamp at the beginning). This guarantees new inserts land sequentially at the end of the tree, eliminating page splits.

  • 2. PostgreSQL's Natural Immunity (Heap Appends):

    Since Postgres is a heap-file system, inserting a random UUID is extremely fast—it simply appends the row to the active table page. Page splits still occur in the Primary Key B+Tree index, but because index leaf nodes only store light 6-byte ctid values (and not the entire row payload), index page splits are far less expensive than InnoDB table page splits.

The Secondary Index Double-Lookup Trade-off

Why do InnoDB secondary indexes store the Primary Key value rather than a direct physical Row ID address? Let's compare their physical pathways during queries:

  • PostgreSQL Direct Lookup Pathway:
    Secondary Index (e.g., email) -> physical ctid (Page 5, Slot 2) -> Direct physical read of Page 5.
    The Performance Benefit: Fast, single-hop reads on secondary queries.
    The Drawback: If Postgres updates a row and changes its physical location, every secondary index must be updated with the new ctid.
  • InnoDB Clustered Double-Lookup Pathway:
    Secondary Index (e.g., email) -> Primary Key value ('user-123') -> Clustered B+Tree traversal -> Actual Row Payload.
    The Performance Benefit: If a row moves physically on disk due to a page split, secondary indexes do not break because they point to the logical key, not the physical address.
    The Drawback: Every secondary index lookup requires a double-traversal of B+Trees, which hurts range scans and secondary queries.

s12. Row-Oriented vs. Columnar Storage (OLTP vs. OLAP)

A table is logically a two-dimensional grid of rows and columns. However, physical storage media (like NVMe SSDs or HDDs) are strictly one-dimensional linear arrays of bytes. To store this table on disk, database systems must choose a serialization orientation: Row-Oriented (optimized for transactions) or Columnar (optimized for analytics).

The Core Structural Difference

The difference lies entirely in how consecutive bytes are organized within the physical storage page:

  • Row-Oriented (OLTP): 1 page containing multiple entire rows. All columns belonging to a specific row are stored contiguously on disk: [Row 1 Col A, Col B, Col C][Row 2 Col A, Col B, Col C].
  • Columnar (OLAP): 1 page containing data for exactly 1 column across thousands of rows. The values for a single column are stored contiguously on disk: Page A: [Row 1 Col A, Row 2 Col A, Row 3 Col A] and Page B: [Row 1 Col B, Row 2 Col B, Row 3 Col B].

OLAP on Row-Oriented: The I/O Bottleneck

When running data-warehousing queries (e.g., aggregations, averages, grouping) on transactional row databases, physical performance degrades catastrophically:

  • Massive I/O Wastage: If you only need to calculate average product prices across 50,000,000 rows, the disk controller must pull every column (user IDs, long string descriptions, seller names, timestamps) into the buffer pool. 90% of the loaded bytes are thrown away instantly.
  • Buffer Pool Pollution: Loading heavy rows to scan just one column evicts valuable transactional pages from RAM cache, slowing down unrelated operations across the application.
OLAP on Columnar: The Speed & Compression Cure

Column-oriented architectures solve data aggregation constraints with absolute disk-level optimization:

  • Direct Sector Sweeps: If you query a single column, the storage engine calculates the exact sector range of that column's files and reads them sequentially. Other column files are never touched.
  • Ultra-High Data Compression: Because an entire page is filled with the same data type (e.g., all 64-bit timestamps or float balances), databases can compress data exceptionally well. Using techniques like Run-Length Encoding (RLE) (storing `[100, 100, 100]` as `3x100`) or Delta Encoding, storage size footprint shrinks up to 90%.
The Columnar Penalty: Row Reconstruction & Inserts

Columnar databases are optimized exclusively for reads. When targeted with writes or transactional lookups, they face extreme performance deficits:

  • The Row Reconstruction Tax: If you execute SELECT * FROM users WHERE id = 42, the engine must open separate physical files/pages for each column, read slot 42 in each file, and stitch the data back together in RAM. This causes heavy random seek latencies.
  • Fragmented Writes: Inserting or updating a single row requires modifying dozens of separate column files simultaneously, causing heavy disk write overhead.
The OLTP Solution: Continuous Append with Page Fills

To support high-throughput, concurrent writes, OLTP engines keep rows contiguous within pages:

  • Atomic Appends: Inserting a row is highly efficient because all columns are bundled together and written inside a single physical page. Only one disk append is executed.
  • Point-Lookups: Traversing an index lands you directly on the Slotted Page where all row columns reside contiguously, ensuring sub-millisecond lookups.
The Workload Division: OLTP vs. OLAP

Because of these physical storage layouts, the industry divides engineering workloads cleanly between storage models:

Storage Metric Row-Oriented Layout (OLTP) Columnar Layout (OLAP)
Primary Workload Online Transaction Processing (high-throughput API writes, updates, lookups). Online Analytical Processing (BI reports, aggregate statistics, data warehousing).
Query Pattern Returns few rows, but reads many/all columns per row. Returns millions of rows, but reads only a few columns per query.
Compression Moderate (due to high variation of adjacent data types within pages). Extremely high (consecutive identical data types enable premium RLE/Delta compression).
Write Characteristics Highly efficient single-row INSERTs and updates. No page-scattering. Very slow single-row writes. Optimized for massive batch file uploads.
Database Systems PostgreSQL, MySQL, SQLite, SQL Server. ClickHouse, Snowflake, Google BigQuery, Amazon Redshift.
